# Permission Policy Template

Use this reference when planning ERP permissions, role-based access control, approval authority, sensitive fields, administrator actions, and audit requirements.

## 1. Permission Policy Overview

| 항목 | 내용 |
|---|---|
| 시스템/모듈 | 예: 구매 요청 관리 |
| 적용 부서 | 예: 전사, 구매팀, 재무팀, 현업 부서 |
| 권한 관리 기준 | 예: 역할 기반, 조직 기반, 프로젝트 기반 |
| 데이터 조회 범위 | 예: 본인, 부서, 법인, 전체 |
| 주요 통제 대상 | 예: 발주 승인, 단가 변경, 지급 요청, 엑셀 다운로드 |
| 권한 변경 승인자 | 예: 시스템 관리자, 부서장, 내부통제 담당자 |
| 정기 점검 주기 | 예: 분기 1회, 반기 1회 |

## 2. Role Definition

| 역할 | 설명 | 대표 사용자 | 주요 책임 | 비고 |
|---|---|---|---|---|
| 요청자 | 업무 요청을 등록하는 사용자 | 현업 담당자 | 요청 등록, 수정, 회수 | 본인 요청만 조회 |
| 검토자 | 요청 내용의 타당성을 검토 | 팀장, 구매 담당 | 검토, 보완 요청 | 금액 기준으로 생략 가능 |
| 승인자 | 업무 확정 권한 보유 | 부서장, 임원 | 승인, 반려 | SoD 적용 |
| 처리자 | 승인된 업무를 실제 처리 | 구매팀, 재무팀 | 발주, 정산, 지급 처리 | 처리 이력 필수 |
| 관리자 | 시스템 설정과 권한 관리 | ERP 관리자 | 코드, 권한, 정책 관리 | 특권 로그 필수 |
| 감사자 | 이력과 통제 준수 검토 | 내부감사, 회계 | 조회, 다운로드 | 수정 불가 |

## 4. Data Scope Rules

| 범위 | 정의 | 적용 예시 | 주의사항 |
|---|---|---|---|
| 본인 | 본인이 생성한 데이터만 조회 | 휴가 신청, 개인 비용 신청 | 대리 등록 시 소유자 기준 필요 |
| 부서 | 소속 부서 데이터 조회 | 팀 비용, 부서 구매 요청 | 겸직/조직 변경 이력 필요 |
| 하위 조직 | 본인 부서와 하위 부서 조회 | 본부장, 사업부장 | 조직도 기준일 필요 |
| 법인 | 동일 법인 데이터 조회 | 법인별 회계/인사 | 다법인 권한 분리 |
| 프로젝트 | 참여 프로젝트 데이터 조회 | 프로젝트 정산, 투입 관리 | 프로젝트 역할 기준 필요 |
| 거래처 | 담당 거래처 데이터 조회 | 영업/CS | 담당자 변경 이력 필요 |
| 전체 | 전체 데이터 조회 | 관리자, 감사자 | 다운로드 통제 필수 |

## 6. Segregation Of Duties

| 통제 항목 | 금지 조합 | 예외 허용 조건 | 로그/승인 |
|---|---|---|---|
| 구매 요청/승인 | 요청자와 최종 승인자 동일 | 소액 전결 정책이 있는 경우 | 전결 사유 기록 |
| 거래처 등록/지급 | 거래처 등록자와 지급 승인자 동일 | 긴급 지급 예외 승인 | 내부통제 승인 |
| 계좌 변경/지급 | 계좌 변경자와 지급 실행자 동일 | 불가 권장 | 변경 이력 필수 |
| 전표 작성/승인 | 작성자와 승인자 동일 | 자동 전표 예외 정책 | 승인 로그 |
| 권한 신청/부여 | 신청자와 부여자 동일 | 불가 | 권한 변경 로그 |
| 마감/마감 해제 | 마감 담당자와 해제 승인자 동일 | 불가 권장 | 해제 사유 필수 |

## 9. Open Questions

- 권한은 조직도 기준인가, 프로젝트/거래처/법인 기준도 필요한가?
- 겸직, 파견, 대리 결재, 휴직, 퇴사자의 데이터 접근은 어떻게 처리하는가?
- 엑셀 다운로드는 모든 조회 가능 사용자에게 허용되는가?
- 민감 필드의 마스킹 해제 권한과 로그 정책은 무엇인가?
- 마감 이후 관리자 수정은 허용되는가?
- 권한 변경은 누가 승인하고 얼마나 자주 점검하는가?
